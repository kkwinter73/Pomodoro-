# hooks-guide — ガードの「なぜ」

ハーネスの心臓部（層3 / §5）。散文の規約を**ツール呼び出し時に効く強制経路**へ昇格させる。
各 hook が block / reminder / 品質 のどの型かと、その選定理由を記す。

## ガードの3型（§5-2）

| 型 | 何をする | 使う条件 | 安全方向 |
|---|---|---|---|
| **block** | 条件に当たれば拒否 | 違反を**機械的に確実判定でき**、止めるべき | 取りこぼし=事故 → 確実判定できる面だけに絞る |
| **reminder** | 止めず注意文を文脈に注入 | 違反かどうか**機械判定できない** | 判断材料を渡すだけ。正規用途を殺さない |
| **品質ゲート** | lint等を多段で回す | コード品質の即時フィードバック | 多段＋空回り防止＋環境差で無音 |

## 配線（`.claude/settings.json`）

| 発火点 | hook | 型 |
|---|---|---|
| PreToolUse(Bash) | `branch-guard.sh` | block |
| PostToolUse(Edit/Write) | `lint.sh post` | 品質（軽量） |
| PostToolUse(Edit/Write) | `decisions-reminder.sh` | reminder |
| Stop | `lint.sh stop` | 品質（フル） |

## branch-guard.sh（block 型）

- **何を**: `main`/`master` 上での `git commit` を拒否する。
- **なぜ block でよいか**: 相手は*悪意ある回避*ではなく*善意のうっかり*（§5-2）。
  「現在ブランチ」と「コマンドに `git commit` が含まれるか」という**構造化された判定**だけで足り、誤検知が出ない面に絞っている。
  取りこぼしても安全側（別経路で commit されても、それは明示操作）。

## decisions-reminder.sh（reminder 型）

- **何を**: `docs/decisions/` への書き込み後、採録基準（「消しても再到達できるなら載せない」）を注意文として注入。
- **なぜ block にしないか**: 「その決定行が本当に恒久・横断か」は**人間でも文脈なしには機械判定できない**（§5-3）。
  block すると正規の追記まで殺すので、**許可した上で判断材料を渡す speed-bump** にしている。

## lint.sh（品質ゲート / §5-4・§5-6）

- **post（編集直後）**: 変更した `*.ts`/`*.tsx` だけを `eslint --quiet` で軽量チェック。重大エラーだけ即フィードバック。
- **stop（ターン終了）**: `npm run lint` のフル検査。コミット前の最後の安全網。
- **無限ループ防止**: Stop hook 入力の `stop_hook_active === true`（=既に1回直しを促した後の再ターン）なら**ブロックせず通知のみ**。直せないエラーで永久ループに陥らせない。
- **環境差で無音**: `node_modules/.bin/eslint` が無い（未 `npm install`）等では exit 0 で素通り（§5-5）。

## 新しいガードを足すときの3問（§5-2）

1. 違反を機械的に確実判定できるか? → No なら **reminder**。
2. 取りこぼしは事故か、確認で済むか? → 事故なら **block を構造化入力に絞る** / 確認で済むなら allow 型で気軽に。
3. 相手は悪意ある回避か、善意のミスか? → 善意なら素朴な検出で十分。
